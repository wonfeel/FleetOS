1. Безопасность моста

Ключ теперь можно не придумывать - генерируется сам

Если запустить мост на адресе, отличном от 127.0.0.1 (например, через FLEET_BRIDGE_HOST для Radmin/LAN), и FLEET_BRIDGE_KEY не задан - мост САМ создаёт случайный ключ, сохраняет его в windows/bridge_key.txt (переиспользуется при перезапусках, не пересоздаётся каждый раз) и печатает прямо в консоль при старте:

[bridge] no FLEET_BRIDGE_KEY set - generated one automatically (saved to
[bridge] ...\windows\bridge_key.txt, reused on every future restart):
[bridge]
[bridge]     SPk55b-Lwv9cu1HzeHdfjqkt7i5pF14k
[bridge]
[bridge] paste it into: `install <url> <key>`, config.lua's apiKey field,
[bridge] or the dashboard's "API key" field.

Остаётся только скопировать этот ключ в команду install, config.lua или поле «API key» в дашборде. Задать свой собственный ключ через FLEET_BRIDGE_KEY по-прежнему можно - автогенерация включается только если вы этого не сделали.

Ключ только для чтения

Раньше был один ключ (FLEET_BRIDGE_KEY) - с ним можно было всё. Теперь можно завести ВТОРОЙ, слабее:

set FLEET_BRIDGE_KEY=полный-ключ
set FLEET_BRIDGE_READONLY_KEY=ключ-только-для-чтения
py windows\bridge_server.py 8787

С FLEET_BRIDGE_READONLY_KEY можно смотреть статус/файлы, но нельзя запускать команды, писать файлы или выполнять /compute. Выдавайте его тем, кому нужно только наблюдать за флотом.

Выполнение compute-скриптов теперь выключено по умолчанию

/compute/<name> запускает произвольный Python/скомпилированный код на вашем ПК - это по сути удалённое выполнение кода. Теперь требует явного включения:

set FLEET_ENABLE_COMPUTE=1
Включайте, только если реально используете windows/compute/*.py (например, вычисления для Raytower) - без него сама фича Raytower и остальной флот продолжают работать как раньше.

Опциональный TLS

Если оба пути указаны, мост поднимется на https:// вместо http://:

set FLEET_TLS_CERT=path\to\cert.pem
set FLEET_TLS_KEY=path\to\key.pem

По умолчанию выключено - обычный http достаточно безопасен для localhost/доверенного VPN. Включайте, если мост смотрит куда-то ещё.

2. Здоровье и метрики

Два новых эндпоинта для мониторинга флота автоматикой (или просто в браузере):

curl http://127.0.0.1:8787/health
curl http://127.0.0.1:8787/metrics   # нужен ключ

/health - без ключа, для healthcheck-проб (сколько узлов известно, сколько из них отчитывались недавно). /metrics - счётчики: сколько команд поставлено в очередь, отчётов получено, ошибок, срабатываний rate-limit.

3. Резервное копирование флота

Раньше состояние флота (папки узлов, очереди команд) можно было бэкапить только вручную копируя bridge_state.json/node_meta.json. Теперь есть normal export/import:

curl http://127.0.0.1:8787/admin/export -H "X-API-Key: полный-ключ" > backup.json

curl -X POST http://127.0.0.1:8787/admin/import -H "X-API-Key: полный-ключ" -H "X-Fleet-Dashboard: 1" --data-binary @backup.json
Import заменяет состояние целиком, не сливает с текущим - используйте для восстановления после сбоя ПК с мостом, не для повседневных операций.

4. Откат ядра/приложения

«Update fleetos» теперь сам делает резервную копию перед обновлением (fleetos.lua.bak). Если новая версия ведёт себя плохо - откатите одной командой:

curl -X POST http://127.0.0.1:8787/command -H "Content-Type: application/json" -H "X-Fleet-Dashboard: 1" -d "{\"node\":\"мой-узел\",\"type\":\"rollback_kernel\"}"

Откатывает ровно на одну версию назад (не полная история) и перезагружает узел - то же самое, что уже было для приложений через rollback, теперь есть и для самого ядра.

5. Массовая настройка флота

Новая команда configure - меняет адрес моста и/или список автозапуска сразу на многих узлах одной командой, вместо ручного редактирования config.lua на каждом:

curl -X POST http://127.0.0.1:8787/command -H "Content-Type: application/json" -H "X-Fleet-Dashboard: 1" \
  -d "{\"node\":\"*\",\"type\":\"configure\",\"bridgeUrl\":\"http://новый-адрес:8787\",\"startup\":[\"clock\",\"fleetbridge\"]}"

bridgeUrl/apiKey и startup независимы - можно передать что-то одно. Новый список автозапуска применится со следующей перезагрузки узла.

6. Разблокировка «зависшего» монитора

Если приложение захватило монитор (fleetos.claimMonitor()) и перестало его отпускать (не крашится, просто больше не перерисовывает) - раньше единственным выходом было убить его через терминал. Теперь можно снять захват удалённо, не трогая само приложение:

curl -X POST http://127.0.0.1:8787/command -H "Content-Type: application/json" -H "X-Fleet-Dashboard: 1" -d "{\"node\":\"мой-узел\",\"type\":\"force_release_monitor\"}"
На практике до этого редко доходит - приложение, которое зависло НАСТОЛЬКО, что не выполняет даже os.pullEvent/os.sleep, теперь само убивается сторожевым таймером ядра раньше, чем понадобится эта команда.

7. Защита Raytower

Трафик Raytower по rednet раньше был совсем без защиты - любой игрок с модемом в радиусе действия мог подделать/переиграть пакеты (подменить решённую позицию, разослать фальшивый опрос). Теперь можно включить подпись:

-- config.lua, ОДИНАКОВО на мастере и КАЖДОЙ вышке:
raytowerSecret = "какой-то-общий-секрет",
Это не настоящий криптографический HMAC-SHA256 - у CraftOS-Lua нет битовых операций, чтобы дёшево собрать настоящий SHA-2. Реализация - keyed-MAC на основе FNV-1a плюс защита от replay по времени: останавливает случайного игрока без секрета, не защищает от целенаправленной криптоатаки. Подробности - game/apps/raytower/_raytower_auth.lua.

Без raytowerSecret всё работает как раньше (без подписи) - это опция, а не обязательное требование, обратная совместимость сохранена.
