Мини-ОС для компьютеров CC:Tweaked (ComputerCraft) плюс Windows-бридж и веб-дашборд для
удалённого управления целым флотом компьютеров, распространяется под лицензией
[MIT](LICENSE). Все компьютеры равны - нет master/slave - и каждый остаётся управляемым,
наблюдаемым и обновляемым из одной вкладки браузера без переключения в игру.

**Проблема:** у компьютера CC:Tweaked есть только свой экран и свой локальный диск - если их
больше пары штук, приходится физически подходить к каждому, чтобы посмотреть, что он делает,
ввести команду или закинуть обновление.
**Подход:** каждый компьютер запускает одно и то же маленькое ядро (`fleetos.lua`) и опрашивает
bridge-сервер на твоём ПК по HTTP; бридж ставит команды в очередь и собирает отчёты;
однофайловый веб-дашборд превращает это в живой терминал, зеркало монитора-периферии, файловый
проводник и метрики - для любого компьютера флота, выбираемого из выпадающего списка.

## Зачем это нужно

У ComputerCraft нет своей многозадачности - программа владеет терминалом, пока не завершится -
и нет встроенного способа увидеть, что делает компьютер, не стоя перед ним. `fleetos.lua` - это
небольшое кооперативное ядро (spawn/kill/list, крах одного приложения не роняет остальные),
которое запускает настоящие "программы" - оболочку, клиент опроса бриджа, зеркало монитора - как
отдельные задачи. Бридж и дашборд существуют, чтобы это ядро было *полезно* уже на масштабе
флота: никто не хочет телепортироваться к 20 разным фермерским компьютерам, чтобы проверить их.

## Что умеет

Без master/slave: каждый узел запускает одно и то же ядро, дашборд просто смотрит на один из них
в данный момент. Узел регистрируется сам, просто один раз опросив бридж. Дальше по списку:

- **Живой терминал** для любого узла - выполняет команду так, будто она введена в реальной
  оболочке (`shell.run`), стримит обратно вывод print/write.
- **Зеркало монитора**: то, что реально нарисовано на подключённой к компьютеру
  monitor-периферии, восстанавливается пиксель-в-пиксель (настоящая палитра CC:Tweaked) в
  браузере, включая нажатие на эмулированный монитор, вызывающее настоящее событие
  `monitor_touch` в игре.
- **Файловый проводник** для файловой системы узла или папок `game/`/`compute/` на ПК. Просмотр,
  редактирование, создание, удаление, деплой перетаскиванием.
- Флотовые массовые операции: обновление/откат с автоматическим бэкапом ядра, пуш конфига (URL
  бриджа, API-ключ, список стартовых приложений), переименование, организация по папкам, макросы.

**Gateway-кластер** (`apps/common/fleetgateway.lua`, см.
[документ по архитектуре](https://wonfeel.github.io/FleetOS/docs/ARCHITECTURE_GATEWAY_CLUSTER.html))
позволяет нескольким доверенным компьютерам ретранслировать poll/report-трафик для остального
флота по `rednet` вместо того, чтобы каждый узел делал свой HTTP-запрос - выбор лидера по
подписанным heartbeat'ам, автоматический failover.

Управление дроном (`apps/drone/`) даёт ручное управление (тяга/рысканье/горизонтальное
перемещение) для 4-моторной наклонно-роторной платформы прямо с дашборда, с непрерывной
стабилизацией положения так, что корпус остаётся горизонтальным независимо от команд пилота, и
failsafe, сбрасывающим тягу в ноль при потере связи. Математика микширования управления покрыта
юнит-тестами на десктопном интерпретаторе Lua (`test/test_motor_mixer.lua`) - сама разводка
redstone/периферии всё ещё требует реальной настройки в игре, см.
[Что ещё не сделано](#что-ещё-не-сделано).

- Триангуляция Raytower вычисляет реальную позицию компьютера по направленным лучам дальномеров,
  репортящимся с башен-компьютеров, с опциональным вычислительным офлоадом на ПК.
- HMAC-подписанный `rednet`-трафик для heartbeat'ов/релея gateway и raytower - опциональный общий
  секрет, защита от replay-атак.
- Локальная симуляция (`windows/craftos_shim.lua`) запускает тот же самый код `fleetos.lua`/
  приложений как процесс Windows, без Minecraft. `windows/run_sim.bat` поднимает бридж плюс два
  симулированных узла и открывает дашборд.

Docker/CI: бридж + дашборд работают в контейнере (`docker compose up`, локальный Python не нужен -
см. заголовок [Dockerfile](Dockerfile) про то, зачем compute-скрипты запускаются внутри него, а не
с полными привилегиями хост-пользователя). GitHub Actions гоняет наборы тестов Python, Lua и
Windows-интеграции на каждый push.

`game/fleetos.lua` - это ядро: кооперативное планирование через корутины, зеркало
monitor-периферии с форвардингом касаний, минимальный pub/sub и shared-state IPC между
приложениями (`fleetos.publish`/`setShared`), watchdog бюджета инструкций, чтобы одно зависшее
приложение не стопорило тик-луп, самообновление с автоматическим откатом при неудачном деплое.

Всё, что выполняется *как задача* под этим ядром, живёт в `game/apps/`: `common/fleetbridge.lua`
(HTTP-клиент опроса, который запускает каждый управляемый узел), `common/shell.lua`,
`common/fleetgateway.lua` (релей-кластер), `raytower/` (master/slave триангуляции), `drone/`
(управление полётом плюс чистая математика микширования, вынесенная в отдельный тестируемый
модуль).

- `windows/bridge_server.py` - Python только на stdlib (`ThreadingHTTPServer`, без зависимостей),
  ~2400 строк, разбит на `FleetState` (данные флота в памяти за одним локом) и `StateRepository`
  (атомарная запись на диск, дебаунсированный поток flush) вместо глобальных переменных модуля.
- `windows/dashboard.html` - единый самодостаточный файл (HTML/CSS/JS, без сборки, без
  фреймворка). Терминал/эмуляция монитора, файловый проводник, таблица флота, метрики и
  управление дроном живут все здесь, с встроенной i18n (EN/RU).
- `windows/craftos_shim.lua` эмулирует ровно тот кусок API CraftOS (`fs`, `os.epoch`, redstone
  I/O, режим `--headless` для CI), которого хватает, чтобы настоящий `fleetos.lua`/приложения
  работали без изменений как процесс Windows - локальная разработка и тесты вообще без Minecraft.

## Метрики: от полного сканирования логов до O(1)

`GET /metrics/history` восстанавливает гистограммы частоты запросов по бакетам (обслуженные
poll'ы, полученные report'ы, rate-limited, HTTP-ошибки, ...) для панели истории в дашборде.
Первая версия перепарсивала весь файл лога на *каждый* запрос, независимо от запрошенного окна:

| Диапазон (ширина бакета) | До | После (лог парсится вживую) | После (полностью в памяти) |
|---|---|---|---|
| минута (5с) | ~600 мс | ~15 мс | нет обращений к логу |
| час (60с) | ~600 мс | ~15 мс | нет обращений к логу |
| 24ч (900с) | ~600 мс | ~500 мс | нет обращений к логу (после 24ч аптайма) |
| неделя (7200с) | ~600 мс | ~500 мс | нет обращений к логу (после недели аптайма) |

"Лог парсится вживую" - читает хвост лога назад и останавливается, как только прошёл запрошенное
окно, вместо того чтобы всегда сканировать весь файл - это доминирующая стоимость для коротких
диапазонов. "Полностью в памяти" - кольцевой буфер на каждый диапазон, обновляемый за O(1) в той
же точке кода, что уже инкрементирует живые счётчики, так что I/O по логу вообще нет, как только
процесс прожил дольше запрошенного окна; фоллбэк на лог покрывает только промежуток до этого
момента.

Строки 24ч/неделя всё ещё показывают ~500мс, потому что ретеншн лога честно ещё не достигает
такой глубины на свежеперезапущенном бридже - это реально читаемые данные, а не впустую
потраченная работа. Кэш на 3-60с (масштабируется по ширине бакета диапазона) не даёт нескольким
открытым вкладкам дашборда каждый раз перетриггеривать это.

## Требования

- **Бридж + дашборд**: Python 3.12+ (только stdlib - для самого бриджа ничего не нужно
  `pip install`), любая ОС, либо `docker compose up` вообще без установленного Python.
- **В игре**: CC:Tweaked (ComputerCraft) в Minecraft, с хостом бриджа, разрешённым в
  `computercraft-server.toml`'s `[http.rules]`.
- **Локальная симуляция без Minecraft**: Windows + интерпретатор `lua` 5.x в `PATH`.
- **Инструменты для разработки/тестов** (опционально): `flake8` (Python lint), `luacheck` (Lua
  lint) - см. [`windows/setup.cfg`](windows/setup.cfg) / [`.luacheckrc`](.luacheckrc).

## Быстрый старт

1. `cd windows && python bridge_server.py` (или `start_bridge.bat`) - поднимает бридж на
   `http://127.0.0.1:8787`. Либо на любой ОС: `docker compose up` (тот же бридж + дашборд, без
   локального Python).
2. Открой этот URL в браузере - это и есть дашборд.
3. На свежем компьютере CC:Tweaked в игре:
   `wget http://<ip-твоего-пк>:8787/install.lua install`, затем `install`.
4. Новый компьютер появится в дашборде через несколько секунд.

Смотри [`docs/quickstart.html`](https://wonfeel.github.io/FleetOS/docs/quickstart.html)
для тех же шагов подробнее, или
[`docs/fleetos_guide.html`](https://wonfeel.github.io/FleetOS/docs/fleetos_guide.html)
для полной картины (настройка бриджа, опциональный API-ключ, устранение неполадок, всё).

## Структура

- [`game/`](game/) - всё, что выполняется *внутри Minecraft*. См.
  [`game/apps/README.md`](game/apps/README.md), чтобы написать своё приложение.
- [`windows/`](windows/) - bridge-сервер, дашборд и локальная симуляция только для Windows.
- [`test/`](test/) - юнит-тесты на Lua, запускаются без Minecraft
  (`cd game && lua ../test/<name>.lua`).
- [`docs/`](docs/) - все гайды для людей; см.
  [`docs/README.html`](https://wonfeel.github.io/FleetOS/docs/README.html) для оглавления.
- [`.github/workflows/ci.yml`](.github/workflows/ci.yml) - Python (Linux + Windows), Lua и
  настоящий end-to-end набор Windows-интеграции, на каждый push.

## Тесты

```
test/test_*.lua (13 файлов)         Тесты ядра/приложений на Lua, без Minecraft
  test_motor_mixer.lua               инварианты математики микширования управления дроном
  test_fleetgateway.lua              выбор лидера + failover
  test_signed_rednet.lua             подпись/верификация HMAC
  ...

windows/test_bridge_server.py (17)  юнит-тесты бриджа (состояние, персистентность, TTL)
windows/test_bridge_server_load.py (5)  конкурентность: 32 потока, уникальные id, без потерь записи
windows/test_integration.py (1)     реальный lua-подпроцесс, полный цикл poll/report/command/result
```

Запустить всё, что гоняет CI, локально:
```bash
cd game && for f in ../test/test_*.lua; do lua "$f" || break; done
cd windows && python -m unittest test_bridge_server test_bridge_server_load test_integration -v
```

## Что ещё не сделано

> [!Warning]
> Математика управления дроном покрыта юнит-тестами, но **никогда не летала** - карта каналов
> redstone, имя периферии гироскопа и все коэффициенты PID в `apps/drone/_drone_config.lua` -
> это отправные точки, требующие настоящей проверки в игре, прежде чем это должно что-либо
> поднимать в воздух.

- Роевое/стайное управление (координация нескольких дронов в стиле boids) - намеренно
  отложено, пока управление одним дроном (ручное + стабилизация положения) не будет реально
  облётано и настроено в игре.
- Миграция на WebSocket - бридж основан на опросе (на практике задержка меньше секунды, но не
  push-based), это потолок отзывчивости ручного управления дроном.
- Автоматическое получение HTTPS/сертификатов - TLS сегодня опционален и настраивается вручную.
  Нормально работает в Radmin LAN, и это честно единственная конфигурация, на которой это
  реально тестировалось.
